# Anthropic《Demystifying evals for AI agents》精读课

原文链接：[Demystifying evals for AI agents](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents)  
发布方：Anthropic  
发布日期：2026 年 1 月 9 日

## 一、先建立直觉

这篇文章想教你的，不是“怎么做一道 LLM 测试题”，而是：

**当模型变成一个会行动的 Agent 后，我们该怎么科学地评估它。**

你先把这两种系统分开：

### 1. 普通 LLM

输入一个问题，输出一段答案。

评测它时，常见做法是：

- 给题
- 看答案对不对
- 打分

### 2. Agent

输入一个任务，它不只是“回答”，还会：

- 多轮思考
- 调工具
- 查资料
- 修改环境状态
- 根据中间结果调整策略

所以 Agent 的评测，难点不在“它说得像不像”，而在：

**它到底有没有真的把事办成。**

这就是全文主线。

## 二、文章最核心的一句话

我帮你把文章压缩成一句学习口诀：

**Agent Eval = 真实任务 + 稳定环境 + 多次试验 + 合适评分器 + 过程复盘**

如果你记住这一句，已经抓到 70% 了。

## 三、文章里的关键术语

这几个词你最好背下来。

### 1. Task

一个任务、一个测试样例。

比如：

- “帮用户退款”
- “修复鉴权漏洞”
- “在浏览器里下单”

### 2. Trial

同一个 task 跑一次，叫一次 trial。

因为模型有随机性，所以同样任务跑两次，结果可能不同。

### 3. Grader

评分器。

它负责判断这次 trial 成功没成功。

### 4. Transcript

完整过程记录。

包括：

- 用户输入
- 模型回复
- 工具调用
- 中间步骤
- 最终输出

### 5. Outcome

最终结果，也就是环境最后真的变成了什么状态。

这是文章特别强调的点。

比如 Agent 最后说：

“已经帮你订好票了。”

这只是 transcript 里的话。

真正的 outcome 是：
**数据库里到底有没有那条订票记录。**

### 6. Evaluation Harness

评测框架/运行基础设施。

它负责：

- 喂任务
- 给工具
- 跑 agent
- 记录 transcript
- 调 grader
- 汇总结果

## 四、为什么 Agent 比普通 LLM 更难评测

文章给出的原因，初学者可以理解成 3 点：

### 1. 它是多步的

前面一步做错，后面会连锁出错。

### 2. 它会改环境

不是只输出文字，而是：

- 改文件
- 改数据库
- 调接口
- 点击网页

### 3. 它可能“聪明到超出你的评分规则”

文章提到一个例子：模型找到了一个更好的做法，但因为评分规则写死了，反而被判失败。

所以文章反复强调：

**不要过度评测路径，要更关注结果。**

也就是：

- 少问“它是不是按我预想的顺序调用工具”
- 多问“它最后有没有达成目标”

## 五、三种评分器，你要真正理解

### 1. 代码型 grader

就是程序规则打分。

常见方式：

- 字符串匹配
- 正则匹配
- 单元测试
- 静态检查
- 检查数据库状态
- 检查工具调用参数

优点：

- 快
- 便宜
- 可复现
- 容易 debug

缺点：

- 容易太死板
- 容易误伤“合理但不标准”的答案

### 2. 模型型 grader

让另一个 LLM 来打分。

适合看这些：

- 有没有同理心
- 表达清不清楚
- 是否覆盖重点
- 语气是否合适
- 是否符合 rubric

优点：

- 灵活
- 能处理开放式任务

缺点：

- 不稳定
- 更贵
- 需要用人工校准

### 3. 人工 grader

让人来评。

优点：

- 最接近真实用户
- 最适合主观任务

缺点：

- 贵
- 慢
- 不容易规模化

### 文章的态度是什么？

**能用代码 grader 就先用代码 grader。**  
**必须灵活时再上 LLM grader。**  
**人工主要用来校准。**

这是很实用的工程观。

## 六、Capability Eval 和 Regression Eval

这是全文非常重要的一个区分。

### 1. Capability eval

问题是：

**“这个 Agent 现在到底能做到什么水平？”**

特点：

- 应该有挑战
- 一开始通过率可以很低
- 用来推动进步

你可以把它理解成：
**爬坡用的题库**

### 2. Regression eval

问题是：

**“它以前会的，现在还会不会？”**

特点：

- 应该接近 100% 通过
- 用来防止退步
- 一旦掉分，说明有回归问题

你可以把它理解成：
**守底线的题库**

### 文章里一个很好的观点

以前用来“测试能力上限”的题，等系统成熟后，可以“毕业”成回归测试题。

这个思路非常像传统软件测试。

## 七、pass@k 和 pass^k

这部分很多新手第一次看会懵，我帮你讲透。

### 1. pass@k

意思是：

**试 k 次，只要成功 1 次就算好。**

适合：

- 多生成几个候选答案没关系
- 只要其中一个能用就行

比如单次成功率 50%，那：

- `pass@1 = 50%`
- `pass@3` 会更高，因为 3 次里碰到一次成功的概率更大

### 2. pass^k

意思是：

**试 k 次，k 次都成功。**

适合：

- 稳定性要求高
- 用户每次都希望它靠谱

文章举的例子：
如果单次成功率是 75%，跑 3 次，
那 `pass^3 = 0.75^3 ≈ 42%`

这说明：

**“偶尔能做对” 和 “稳定能做对” 完全不是一回事。**

### 你怎么选？

- 写代码候选解、生成多个备选方案：更适合看 `pass@k`
- 客服、金融、面向用户的 Agent：更该看 `pass^k`

这是很重要的产品视角。

## 八、不同 Agent 怎么评测

文章按类型讲了几类 Agent，我帮你提炼成初学者版。

### 1. Coding Agent

比如修 bug、改代码、跑测试。

重点评测：

- 测试是否通过
- 代码是否正确
- 有没有引入新问题
- 工具使用是否合理

最自然的方法：
**单元测试 + 静态检查 + 必要时用 LLM 看代码质量**

因为代码通常有比较明确的“对/错”。

### 2. Conversational Agent

比如客服、销售、辅导。

重点评测：

- 问题有没有解决
- 语气是否得体
- 是否验证身份
- 是否调用了该用的工具
- 是否在合理轮数内完成

这里通常要同时看：

- `outcome`：事情办成没
- `interaction quality`：沟通质量好不好

### 3. Research Agent

比如调研、资料汇总、写报告。

难点在于：

- “完整”很主观
- 信息源会变化
- 输出开放性强

所以常见评测组合是：

- 事实是否有依据
- 关键点是否覆盖
- 来源是否可靠
- 报告是否连贯清晰

### 4. Computer Use Agent

比如在网页或桌面环境里点按钮、滚动、输入。

重点评测：

- 最终页面/系统状态是否正确
- 后端状态是否真的发生变化
- 它有没有选对交互方式

文章举了很好的例子：

- 看 Wikipedia，总结内容时，DOM 提取更高效
- 在 Amazon 找商品，截图交互可能更划算

所以评测的不只是“做没做完”，还包括：
**是不是用对了工具策略。**

## 九、从 0 到 1 怎么建 eval

这部分最适合你以后自己做项目时直接照着用。

### Step 0：尽早开始

文章建议：
**20 到 50 个真实任务，就已经是很好的起点。**

不要等“题库很大很全”才开始。

### Step 1：先把你已经手工测的东西收集起来

比如：

- 每次上线前你都会试的功能
- 用户常见投诉
- Bug tracker 里的真实问题

把这些变成 eval task。

### Step 2：任务必须写清楚，而且要有参考解

好任务的标准是：

**两个领域专家独立看这个任务，应该会得出差不多的 pass/fail 结论。**

文章还提醒了一个特别实用的坑：
如果你让 agent “写个脚本”，却没说脚本放哪，
而 grader 又默认它必须在某个路径，
那失败可能不是 agent 的错，而是题目有歧义。

所以要有：

- 明确任务描述
- 明确 success criteria
- 明确 reference solution

### Step 3：题集要平衡

不能只测“该做时会不会做”，也要测“不该做时会不会乱做”。

文章举的是搜索例子：

- 应该搜索的问题
- 不该搜索、应该直接回答的问题

都要测。

否则模型可能会被训练成“逢题必搜”。

### Step 4：环境要稳定，而且每次 trial 要隔离

每次运行最好从干净环境开始。

否则会出现：

- 上一次留下的文件影响这一次
- 缓存导致虚高成绩
- 资源不足导致一串任务一起失败

文章甚至提到内部曾发现模型利用前一次 trial 的 git history 占便宜。

这说明环境污染是真问题。

### Step 5：设计 grader 要聪明

文章的建议非常值得记住：

- 尽量用 deterministic grader
- 必要时再用 LLM grader
- 不要死盯“它是不是按某条路径做”
- 更关注“最终产物和结果”
- 复杂任务要允许 partial credit
- LLM grader 要给它“不确定就说 Unknown”的出口
- LLM grader 要用人工校准

### Step 6：一定要读 transcript

这是全文反复强调的重点。

因为低分可能不是 agent 真差，而是：

- grader 有 bug
- task 有歧义
- harness 限制了 agent
- agent 其实找到了一条合理新路径

### Step 7：注意 eval 饱和

如果一个 eval 已经 100% 通过，
它就不再适合衡量“还能不能继续进步”，只能用来防回归。

### Step 8：长期维护

文章强调：
**eval suite 是活的，不是一次性文档。**

应该像单元测试一样持续维护。

## 十、这篇文章最深的一层思想

如果你只记工具和术语，其实还没真正学会。

这篇文章更深的一层思想是：

**评测本身也需要被评测。**

也就是说，当分数不好时，你不能立刻下结论“模型不行”。

你要同时怀疑 4 件事：

- 是不是模型真差
- 是不是 task 写坏了
- 是不是 grader 写坏了
- 是不是 harness/环境限制了 agent

这是一种很成熟的工程思维。

文章里提到一个例子：
某个 benchmark 上模型起初分数很低，
后来研究人员检查发现问题包括：

- 评分过于死板
- 任务规格含糊
- 随机性无法复现
- scaffold 过于受限

修完后分数大幅上升。

这说明：
**低分不一定是能力低，也可能是评测设计低。**

## 十一、eval 不是全部，它只是其中一层

文章最后讲得很完整：

理解 Agent 质量，不能只靠 automated evals，还要配合：

- 生产监控
- A/B 测试
- 用户反馈
- 人工读 transcript
- 系统化人工研究

它用了一个“瑞士奶酪模型”的类比：
每一种方法都有洞，但多层叠加后，漏洞更不容易漏过去。

这个观念非常重要。

## 十二、你现在应该能回答的 5 个问题

如果你真的学到了，下面这 5 题你应该能答出来：

1. 为什么 Agent 的评测比普通问答模型更难？
2. transcript 和 outcome 有什么区别？
3. code-based grader 和 model-based grader 分别适合什么？
4. capability eval 和 regression eval 有什么不同？
5. pass@k 和 pass^k 为什么会得出完全不同的结论？

## 十三、给你的“初学者版总结”

你可以把整篇文章压缩成下面这段话背下来：

**Agent eval 的本质，是用真实任务在稳定环境中多次运行 agent，再用合适的 grader 判断它是否真正完成目标，并通过读 transcript 来确认分数可信。**  
**评测不只是在测模型，也是在测任务设计、评分逻辑和运行环境。**  
**优秀团队会同时做 capability eval、regression eval，并把 automated eval、生产监控、用户反馈和人工复盘结合起来。**
